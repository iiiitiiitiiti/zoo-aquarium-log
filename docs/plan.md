# 動物園・水族館 訪問記録アプリ 実装プラン

> **ステータス（2026-07-17）: ステップ1〜11 すべて実装完了。** 施設マスタは全47都道府県 473施設（ステップ7完全終了・2026-07-16）。本書は設計判断の記録として保持し、以後の変更は個別プラン（docs/archive/ 参照）と定期監査で管理する。当初の要件からの主な拡張: 訪問写真の添付（当初「写真なし」→ Firebase Storage で実装）、施設メモの複数登録・Markdown 表示。

## Context

全国の動物園・水族館（JAZA 非加盟も含め広く網羅、推定 300〜500 施設）をリストアップし、家族で訪問記録をつけるアプリを作る。ヒアリングで確定した要件：

| 項目 | 決定内容 |
|---|---|
| 利用形態 | スマホ中心の Web アプリ（PWA 対応、PC でも閲覧可） |
| 利用者 | 世帯で1つの記録を家族全員で共同編集 |
| 認証 | 合言葉方式（各自のアカウント登録は不要）。厳密なセキュリティは求めない |
| 施設範囲 | 非加盟含め広く網羅。現存施設のみ（将来閉園したら「閉園済み」ステータスで記録保持、制覇率の分母から除外） |
| 記録内容 | 訪問日・メモ感想・評価（星）。写真なし。同一施設への複数回訪問を個別記録 |
| 機能 | 地図表示 / 都道府県別制覇率・統計 / 行きたいリスト（未訪問向け）/ お気に入り（訪問済みの推し施設）/ 施設の手動追加 |
| 初期データ | Claude が Web 調査して整備 |
| 技術 | おまかせ |

## 採用構成（案比較の結論）

**静的施設マスタ JSON ＋ Firebase Firestore ＋ Vite + React + TypeScript の PWA、GitHub Pages ホスティング**

- **BaaS = Firebase Firestore（Spark 無料枠）**: Supabase 無料枠は7日間非アクティブでプロジェクトが休止する（数週間に1回の利用想定では致命的）。Firebase Spark は休止なし・50k reads / 20k writes/日・1GB で家族利用には十分（2026-07 Web 検索で確認済み）。
- **施設マスタはアプリ同梱の静的 JSON**: マスタは Claude が整備しコードと一緒にバージョン管理。Firestore квота を消費せず、マスタ更新はデプロイで配信。手動追加施設のみ Firestore に保存。
- **フロント = Vite + React + TS**: リスト・地図・統計・共同編集の状態管理があるためフレームワーク採用。地図は Leaflet + OpenStreetMap（無料・API キー不要・帰属表示必須）＋ marker clustering。
- **ホスティング = GitHub Pages**: 既存サイトと運用を揃える。GitHub Actions でビルド→デプロイ。
- 不採用: Supabase（休止問題）、Oracle VM 自前 API（保守負担・SSH 鍵が Mac のみ）、素の HTML/JS（この規模では状態管理が破綻）、Cloud Functions での合言葉検証（家族数人の趣味アプリには過剰）。

## 認証設計（Codex レビューを受けて改訂）

**合言葉の UX はそのまま、実体は Firebase Auth の世帯共有アカウント**とする。

- アプリに固定メールアドレスを埋め込み、初回に合言葉（6文字以上）を入力 → `signInWithEmailAndPassword(固定メール, 合言葉)` でサインイン。認証状態は Firebase SDK が永続化し、以後入力不要。
- Firestore ルールは `request.auth.uid == 世帯アカウントのUID` で判定。**UID ベースなので推測・列挙が構造的に不可能**になり、総当たりも Firebase Auth のレート制限で抑止される（当初案の SHA-256(合言葉) capability 方式は、弱い合言葉だと辞書攻撃可能なため破棄）。
- 合言葉の変更 = Auth のパスワード変更で対応可能。合言葉を忘れた場合はメールでリセット可能（固定メールは自分の Gmail エイリアス等にする）。
- ルールは包括ワイルドカード許可を避け、`visits` / `marks` / `customFacilities` を個別に match。collection group query 用ルールは作らない。create/update 時に型・文字数上限・評価範囲（1〜5）・許可フィールドを検証。
- **Emulator Suite で負例テスト**を書く: 未認証アクセス / 他 UID / collection group query / 不正スキーマ / 過大メモ、がすべて拒否されること。

## データモデル

**静的 `facilities.json`（アプリ同梱）**
```
{ id,                  // 永続ID。名称変更・閉園でも再利用しない
  name, kana, pref, city,
  type: "zoo" | "aquarium" | "both" | "other",
  lat, lng,            // 来園者入口または施設代表点と定義
  url,                 // 公式サイト
  sourceUrls: [],      // 調査時の一次情報URL（必須）
  status: "open" | "closed" | "suspended",  // 閉園は削除せずstatus変更
  lastVerifiedAt }     // 最終確認日
```
- 動物園＋水族館の併設複合施設は「入場券が同一なら1件（type: both）、別料金・別入場なら2件」を原則とする（調査開始時に基準案として再確認）。
- 手動追加施設の ID は `custom_` プレフィックスで名前空間を分離。マスタと重複が判明しても自動統合せず表示上の注意に留める。

**Firestore `households/{hid}/`**（hid = 世帯アカウント UID）
- `visits/{visitId}`: `{ facilityId, date, rating?, memo?, visitor?, createdAt, updatedAt }`
  - 複数回訪問は複数ドキュメント。**visitId はクライアントで生成し `setDoc` で保存**（送信リトライ・二重押下を冪等化）。`createdAt`/`updatedAt` は `serverTimestamp()`。同施設・同日の重複は警告のみ（禁止しない）。
- `marks/{facilityId}`: `{ wishlist: bool, favorite: bool }` — フィールド単位更新。同時トグルは last-write-wins を仕様として許容。
- `customFacilities/{custom_id}`: facilities.json と同スキーマ。

統計（制覇率・推移）は保存せず、visits からクライアント側で派生計算する（集計カウンター文書を持たない → 同時編集の競合源を作らない）。

## 画面構成

1. **施設リスト**（ホーム）: 検索、都道府県・種別・状態（訪問済/未訪問/行きたい/お気に入り）フィルタ
2. **施設詳細**: 施設情報、訪問記録の一覧・追加・編集・削除、行きたい/お気に入りトグル
3. **地図**: Leaflet + OSM、訪問状態で色分けピン、クラスタリング、OSM 帰属表示
4. **統計**: 全体制覇率、都道府県別 x/y 館、種別別、訪問数の推移
5. **設定**: 合言葉（パスワード）変更、施設の手動追加、**JSON エクスポート**（家族の長期記録のバックアップ手段として早期に実装）

ルーティングは当面**ルーターなし（タブ切り替え）**とし、必要になったら HashRouter を採用（GitHub Pages は任意パスを index.html に rewrite しないため BrowserRouter は直リンクで 404 になる）。

## GitHub Pages / PWA の注意点（ハマりどころを事前明記）

- Vite の `base` を `/<リポジトリ名>/` に設定。manifest の `start_url`・`scope`・アイコン URL・Service Worker 登録パスもこれに一致させる。
- Firebase の API キー等の設定値はクライアント公開前提（秘密情報ではない）。保護はセキュリティルールで行う。App Check は趣味アプリには過剰として見送り。
- Service Worker 更新時の旧キャッシュ残留対策として「新バージョン検知→自動リロード」方針を採る。
- OSM タイルは precache しない。オフライン保証範囲は「アプリシェルの起動まで」と定義（Firestore 未取得データのオフライン閲覧は保証しない）。

## 施設マスタ整備の進め方（品質担保）

1. **包含・除外基準を先に確定**（ふれあい牧場・昆虫館・移動水族館などの線引き。基準案を提示してユーザー確認）
2. **代表 20〜30 施設でパイロット**: スキーマ・重複検出・座標検証・複合施設の扱いを検証器（再実行可能なスクリプト）ごと確立
3. その後に都道府県ブロック別のサブエージェント fan-out で全件調査
4. 機械検証: スキーマ検証 / 座標が対象都道府県内かのチェック / 正規化名称＋住所＋近接座標での重複候補検出 / 全件の sourceUrls 到達確認
5. 収集者とは別工程での二段階チェック、曖昧な候補のみ人手確認
6. 成果物は「facilities.json ＋ 検証スクリプト ＋ 差分レビュー」

## Codex レビュー

2026-07-13 に codex-skill（gpt-5.6-sol）でレビュー実施。全体構成（静的マスタ＋Firestore＋Vite/React/GitHub Pages）は「修正後なら妥当」との評価。

**反映した主な指摘**:
- SHA-256(合言葉) capability 方式は弱い合言葉への辞書攻撃が可能 → **世帯共有の Firebase Auth アカウント方式に変更**（合言葉 UX は維持）
- 包括ワイルドカードのルール許可は collection group query で世帯横断アクセスを許す恐れ → サブコレクション個別 match ＋ Emulator 負例テストを工程化
- 施設 ID のライフサイクル規約（永続 ID・status 管理・sourceUrls・lastVerifiedAt・custom_ 名前空間）を明文化
- 全件 fan-out 前に代表 20〜30 施設のパイロット＋再実行可能な検証スクリプトを先行
- GitHub Pages の base/manifest/SW スコープ一致、BrowserRouter の 404 問題 → ルーターなし方針を明記
- 訪問追加のクライアント生成 ID ＋ setDoc による冪等化、serverTimestamp 使用
- JSON エクスポート（バックアップ）を機能に追加
- 実装順を「基準→パイロット→雛形→Auth/Rules→CRUD→全件整備→…」に再編

**見送った指摘と理由**:
- App Check の導入: 家族数人の趣味アプリには過剰。ルール＋Auth で十分と判断
- `mergedInto` / `masterVersion` フィールドの必須化: 施設統合はレアケース。重複統合が実際に必要になった時点で導入（YAGNI）
- Cloud Functions での合言葉 KDF 検証（案C）: Codex 自身も「過剰・非推奨」としており不採用
