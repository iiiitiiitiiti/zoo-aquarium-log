# zoo-aquarium-log ステップ7: 施設マスタ全件整備（150→300〜500件）

## Context

`docs/plan.md` の実装ステップ7。合意済みの実装順（6→10→7→8→9→11）で、ステップ6は実装済み・ステップ10はプラン承認済み。現在の施設マスタは150件で**JAZA正会員の網羅は完了**しており、残るは**JAZA非加盟施設**の調査・収録。300〜500件は**目標ではなく推定**であり、最終件数は調査結果として報告する。

- 過去実績: JAZA拡充時「公式リスト抽出→既存突合→未収録98件を Sonnet 5施設×20班・4ウェーブで一次情報調査→座標は国土地理院APIで補完→全項目検証」が成功

## Phase C: 検証スクリプト拡張（唯一のコード変更）

`validate-facilities.mjs` の既存スタイル（同期・純関数）を踏襲。既存 `validateFacilities` のシグネチャは変えない。

1. **座標の地理チェック** — エラーと警告の二層に分ける（バウンディングボックスを「都道府県内」の確定判定に使わない）:
   - **エラー（`test:data` 組み込み）**: 座標が**日本の全域バウンディングボックス**（本土＋離島を含む粗い範囲）の外にある場合のみ。緯度経度の取り違え・符号ミス・海外座標という明白な異常を機械で止める
   - **警告（新規 `scripts/check-geo.mjs`・レポート出力）**: 47都道府県バウンディングボックス（新規 `scripts/pref-bounds.mjs`）で `inside` / `outside`（要確認）/ `manual-review`（県境・離島近接）の3値判定。**outside でも即座に拒否せず人手確認に回す**（ボックスは県境を厳密になぞれないため、確定エラーにすると県境の正当な施設を誤って弾く）
   - `pref-bounds.mjs` には**出典・作成日・座標系をコメントで記録**。テーブル健全性のテスト: 47県網羅・min<max の範囲整合・各ボックスが `prefectures.ts` の県庁所在地を含む・代表的な離島（伊豆諸島／隠岐／対馬／五島／奄美／宮古八重山）の座標が該当県ボックス群に入る固定フィクスチャ
   - 導入時に既存150件で警告ゼロ（または警告の個別解消）を確認
   - 行政区域ポリゴン（国土数値情報）による point-in-polygon は今回見送り（後述の見送り理由参照）
2. **重複候補検出**（新規 `scripts/find-duplicate-candidates.mjs`・警告レポート）:
   - 名称正規化（全半角・空白・種別語ゆらぎ吸収）＋同一 pref 近接座標（数百m）または名称高類似
   - 人間が判断するためのペア一覧。自動統合しない。Phase A の既存150件との突合にも流用
   - `npm run check:duplicates`
3. **sourceUrls 到達確認**（新規 `scripts/check-source-urls.mjs`・`npm run check:links`）:
   - `test:data` から分離（外部ネットワーク依存の隔離）
   - **結果は二値でなく多値分類**: `reachable` / `redirected` / `not_found`(404/410) / `blocked`(401/403) / `rate_limited`(429) / `server_error` / `timeout` / `network_error`
   - **`not_found` のみ強い失敗**扱い（URL差し替え必須）。`blocked` / `rate_limited` は「未検証」として人手確認へ回す（bot対策サイトの誤判定防止）。bot 対策の回避目的の大量リトライはしない
   - ユニークURL集合に対して実行し、**ドメイン単位のレート制限**（同一ドメインへ連続アクセスしない）を入れる

## Phase A: 候補列挙＋カバレッジ台帳

- **情報源の二段構え**: 候補列挙は二次情報可（Wikipedia 一覧/カテゴリ、都道府県・市区町村の観光公式、観光系サイト、ファン運営DB、汎用検索）。**収録判定と sourceUrls は一次情報必須**
- **ブロック分割**: 基本は8地方区分=1 Sonnet エージェントだが、**候補が多い関東・近畿は県単位に分割**（1エージェントの出力超過・欠落を防ぐ）。離島散在県は県ごとに検索クエリを変える指示を明記
- **成果物は固定形式の候補台帳**（スクラッチパッド上の JSON。チャット本文での受け渡しはしない）: 候補ごとに `name / pref / city / 発見元URL / 種別見込み / 既存突合結果 / 処置(新規調査・収録済み・除外・要目視) / 除外理由`
- **県別カバレッジ台帳**: 47都道府県すべてについて「当たった情報源（観光公式・Wikipediaカテゴリ・汎用検索の3系統以上）と結果」を記録。**候補0件の県も「0件である根拠」を記録**する。施設種別ごとに検索語を使い分ける（動物園／水族館／サファリ／昆虫館／ふれあい牧場…）
- **停止基準**: 「新規候補が出なくなったら終了」ではなく、(1) 全47県のカバレッジ台帳が埋まっている、(2) 発見候補が全件いずれかの処置に整理されている、(3) **異なる検索戦略で2巡して新規候補ゼロ** — の3条件で列挙完了とする

### Gate 1（必須の停止点）

候補列挙完了時に**必ず停止してユーザー承認を得る**。報告: 新規候補総数（都道府県別内訳）／カバレッジ台帳の要約／想定ウェーブ数・班数／トークン消費見込み（JAZA 98件=20班・4ウェーブの実績から単純比例で、300件超なら60〜70班相当＝**過去最大規模**である旨を明示）／曖昧カテゴリの概算。範囲を絞る判断（小規模個人経営施設の扱い等）はここでユーザーに委ねる。

## Phase B0: 校正ウェーブ（パイロット）

本格 fan-out の前に**非加盟施設 20〜30件（4〜6班）だけ先行調査**し、結果をメインが全件レビューして指示テンプレート・出力形式・判定のブレを修正する（plan.md の「パイロット必須」原則を非加盟施設の調査手順にも適用）。ここで手順が安定してから Phase B の残りへ進む。

## Phase B: 調査ウェーブ

- 5施設×N班（Sonnet）を1ウェーブ、**同時5〜6並列まで**（同一サイトへのアクセス負荷配慮）。地方ブロックごとに調査→レビュー→マージ→コミットまで完結させてから次ブロックへ
- 1施設の調査手順（指示テンプレートに `facility-criteria.md` 全文を埋め込む）:
  1. **基準適合判定を最初に**。明確な除外対象は即ドロップ（理由を台帳に記録）
  2. 全フィールド収集（name/kana/pref/city/type/url）
  3. 座標裏取り: 国土地理院API・公式サイト住所・Wikipedia infobox のクロスチェック。入口または代表点。**座標の根拠を台帳に記録**
  4. sourceUrls: 一次情報を最低1件、可能なら2件
  5. **status 現況確認**: 「施設名 閉園」「施設名 休園」で確認検索
  6. **ID は「提案」まで**（確定しない。確定は Phase D の中央工程 — ローマ字表記揺れ・衝突を1箇所で解消するため）
  7. lastVerifiedAt = 調査実施日
- **出力は固定JSON形式**（facilities.json スキーマ＋台帳項目: 調査日時・座標根拠・採用根拠URL）。チャット散文での納品は不可
- **判定保留**: 曖昧ケースは facilities.json に入れず保留リストへ（名称・所在地・URL・迷った理由）

## Phase R: 独立レビュー（二段階チェック）

plan.md が工程化している「収集者とは別工程での二段階チェック」を機械検証で代替せず、独立のエージェント再調査として実施する:

- **100%再確認**: 判定保留・県境/離島（check-geo の manual-review/outside）・閉園/休園と判定した施設・重複候補ペア・`blocked` 判定の sourceUrls
- **層化サンプリング 10〜20%**: 残りの収録分から県・種別・調査班で層化して抽出し、**収集者とは別の Sonnet エージェント**が一次情報を再調査して突合
- **不一致率が閾値（例: 10%）を超えた班は全件再確認**
- 結果は Gate 2 でユーザーへ報告（保留裁定と合わせて）

### Gate 2

判定保留リスト＋独立レビューの不一致案件をユーザーが裁定（収録／除外／継続保留）。量が多ければ地方ブロック逐次レビューに切替。

## Phase D: 中央マージ工程

**ID確定・重複確定・並べ替え・マージを単一の中央工程（メイン）に集約**する（調査エージェント任せにしない）:

1. ID提案の表記揺れ統一・衝突解消 → 確定
2. `check:duplicates` 実行 → 高確度重複の解消
3. 都道府県コード順・県内かな順の該当位置へ挿入（決定論的な並べ替え。既存部分の並びを変えない）
4. `npm run test:data` ＋ `node scripts/check-geo.mjs` → 警告の個別解消
5. ブロック単位でコミット（例: `feat: 施設マスタに東北地方の非加盟施設N件を追加（150→M件）`）

## 完了の定義（Gate 3）

1. **件数レポート**: 最終件数・都道府県別内訳（佐賀0件・1件県10県の解消状況）・type/status 分布・増分・カバレッジ台帳の全県完了
2. **機械検証**: `test:data` 全パス／`check-geo` の警告全件解消（inside 化 or 人手確認済みの記録）／`check:duplicates` 高確度0件／`check:links` 全件実行で `not_found` 0件・`blocked` は人手確認済み
3. **独立レビュー完了**: 100%対象の再確認済み＋層化サンプリングの不一致率が閾値以下
4. **地図目視の代替**: スクラッチパッドに使い捨て Leaflet+OSM 単発HTML（CDN・コミットしない）で全件を type 別色分けプロットし目視（境界ぎわの取り違えの最後の砦）
5. **判定保留の決着**: 全件がユーザー裁定済み
6. 地方ブロックごとのコミットが `git log` で追える

## リスクと注意点

1. **完全網羅は不可能** — カバレッジ台帳＋2巡ゼロの停止基準で線を引き、Gate 1 で「最終件数は前後する」ことを明示
2. **二次情報の鮮度** — Phase B の現況確認検索＋Phase R の独立レビューで軽減（100%は防げない）
3. **トークン消費が過去最大** — Gate 1 承認＋ブロック逐次進行＋Phase B0 の校正で暴走を防ぐ。独立レビュー分（+10〜20%）も見込みに含める
4. **バウンディングボックスの限界** — 確定判定に使わない（警告＋人手確認）。境界精度は地図目視が最終防衛線
5. **表記ゆれによる重複再収録** — 機械突合＋重複検出＋中央マージ工程＋独立レビューの四重で対応
6. **調査エージェントの出力形式ブレ** — 固定JSON台帳＋Phase B0 での校正で抑える

## Codex レビュー

2026-07-14 に codex-skill（gpt-5.6-luna, reasoning effort: high）でレビュー実施。「方向性は妥当だが、このまま実行開始は早い。4点が修正必須」との評価。Codex 自身も現物確認を行い、150件・46都道府県・`test:data` 7件通過を裁定の根拠にしている。

**反映した指摘**:
- **バウンディングボックスを都道府県の確定判定に使わない** → エラーは「日本全域ボックス外」（緯度経度取り違え等の明白な異常）のみとし、都道府県ボックスは inside/outside/manual-review の3値**警告**レポート（`check-geo.mjs`）へ変更。県境の正当な施設を誤って弾くリスクを排除
- **境界フィクスチャの追加** → ボックステーブルのテストに範囲整合・県庁所在地包含・代表離島の包含チェックを追加。テーブルに出典・作成日を記録
- **停止基準「3系統で新規なし」は網羅性の根拠として弱い** → 47県×情報源の**カバレッジ台帳**（0件県は0件の根拠も記録）＋候補全件の処置整理＋**異なる検索戦略2巡ゼロ**に変更。300〜500件は目標でなく推定と明記
- **二段階チェックの欠落**（plan.md が工程化していた「収集者とは別工程」を機械検証で代替していた） → **Phase R（独立レビュー）を新設**。高リスク案件（保留・県境離島・閉園・重複・blocked URL）は100%、残りは層化10〜20%を別エージェントが再調査、不一致率閾値超の班は全件再確認
- **URL到達確認の二値判定は誤判定を生む** → 多値分類（not_found のみ強失敗、blocked/rate_limited は人手確認へ）＋ドメイン単位レート制限に変更
- **ID採番の分散はローマ字揺れ・衝突を生む** → 調査エージェントは提案まで、**確定は Phase D の中央マージ工程に集約**
- **候補の受け渡し形式が未定義** → 固定JSON形式の候補台帳・監査項目（発見URL・除外理由・座標根拠・調査日時）を必須化
- **関東1エージェントは出力超過リスク** → 候補の多い関東・近畿は県単位に分割
- **20〜30件での手順再校正** → Phase B0（校正ウェーブ）を新設

**見送った指摘と理由**:
- **行政区域ポリゴン（point-in-polygon）の導入**: 国土数値情報の行政区域データの取得・変換・軽量化はそれ自体が1タスク分の重さで、家族向け趣味アプリの座標検証には過剰。ボックス警告＋地図目視＋独立レビューの組で十分と判断。ステップ8（地図画面）実装後は実画面での目視が可能になるため、それでも問題が出た場合の将来課題とする
- **並べ替えスクリプトへの懸念**: 「使い捨てスクリプト任せにしない」の趣旨は中央マージ工程への集約（採用済み）で満たしている。並べ替え自体は決定論的な処理であり、既存部分の並びを変えないことをマージ時チェックで担保する
